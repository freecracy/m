# 客户端缓存

当 tracking 开启时,服务端会记住每个客户端请求的 key , 当 key 值发现变化时会通过 RESP3 协议发送一次(仅一次)失效消息给客户端

CLIENT TRACKING ON|OFF [REDIRECT client-id] [PREFIX prefix] [BCAST] [OPTIN] [OPTOUT] [NOLOOP]
1
参数 作用
redirect 将失效消息转发到另一个客户端,一般使用老的 RESP2
bcast 使用广播模式,一般需要设置 key 前缀
prefix 模式匹配 key 前缀
optin 当广播模式没有激活时,正常不会 track 只读的 key,除非在 client caching yes 之后被调用
optout 当广播模式没有激活时,正常会 track 只读key, 除非在 client caching off 之后被调用
noloop 不发送更新这个 key 的客户端

# 变慢了

  1. 慢日志 : 如命令 sort、sunion、zunionstore
  2. 存储大 key : 如果一个 key 吸入数据非常大,分配内存时也会耗时,删除时也会耗时
  3. 是否有 key 集中过期 : 过期删除策略不会记录在慢日志中,只能从业务中查找
  4. 内存上限 : 是否在执行内存淘汰策略
  5. 是否在执行备份或主从同步,生成 rdb 文件等 : 持久化要执行 fork 操作, fork 执行过程中,父进程需要拷贝内存页给子进程,如果内存占用大拷贝内存页会非常耗时会消耗大量 cpu, fork 完成前实例会阻塞不处理请求. info 命令可以查看最后一次 fork 耗时 lastest_fork_usec 单位毫秒
  6. aof 策略
  7. 是否使用 swap 分区 : 需要对内存和 swap 使用情况监控,对内存不足和使用 swap 时及时报警.
  8. 网卡负载过高 : 对机器各项指标增加监控,包括网络流量

# 多线程

我们说的单线程是指 redis 接收连接、接收数据并解析协议发送结果等命令的执行都是在主线程中执行的.

  • redis 瓶颈不在 cpu,而在内存和网络 io
  • 使用单线程可以简化数据库结构设计
  • 可以减少多线程锁带来的性能损耗

当 socket 中有数据时,redis 会通过系统调用将数据从内核态拷贝到用户态,供 redis 解析用,这个拷贝过程是阻塞的,即同步阻塞 io,数据量大时拷贝延迟越高,解析协议时间消耗越大,这些操作都是在主线程中处理的,特别是连接数过多时情况更加明显,所以提出了多线程,即将接收与发送数据使用多线程并行处理,降低主线程的等待时间.

# 设计思路

  1. 创建一组大小为 io 线程个数的等待队列,用来存储客户端网络套接字
  2. 平均分配客户端网络套接字到等待队列
  3. 等待线程组接收协议完毕或发送数据完毕
  4. 执行后序操作,然后跳到 2 继续执行

# 小结

  1. 个数设置为 1,实际上还是只使用主线程
  2. 默认只负责 write,即发送数据给客户端
  3. 个数一旦设置,不能通过 config 动态设置
  4. 设置了 ssl 后, io-threads 将不工作
  5. io 线程只能读或者写,不存在读写并行的情况

# memory

info memory
1
  1. used_memory : 内存分配器分配的内存总量,以字节为单位,包含实际缓存占用内存和自身运行所占的内存(元数据、lua),是内存分配器分配的内存,这个数据并没有把内存碎片浪费的内存统计进去.
  2. used_memory_human : 同 used_memory,单位时 M
  3. used_memory_rss : 从操作系统上显示已分配的内存总量,单位字节
  4. mem_fragmentation_ratio : 内存碎片率
  5. used_memory_lua : lua 脚本引擎所使用的内存大小
  6. mem_allocator : 在编译时指定的内存分配器,可以是 libc、jemalloc、tcmalloc

# 因内存交换引起的性能问题

当 used_memory 大于可用最大内存操作系统开始进行内存与 swap 空间交换,把内存中旧的或不再使用的内容写入硬盘.

# slave 删除过期key

当一个 key 要过期时,master 会先清除它,之后 master 会向 slave 发送一个 del 命令,slave 删除这个 key 达到主从一致.在 slave 上执行 randomkey 未避免长时间找不到符合条件的key, slave 最多在 hash 表中寻找 100 次无论能否找到都会退出循环.
