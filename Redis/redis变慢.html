# redis 变慢

# 基准测试

确认网络原因还是 redis 真的变慢,在服务器 redis 服务器测试响应延迟.

redis-cli -h 127.0.0.1 -p 6379 --intrinsic-latency 60 # 60s 内最大响应延迟
redis-cli -h 127.0.0.1 -p 6379 --latency-history -i 1 # 最大最小平均访问延迟,每 1s 菜样

  1. 命令时间复杂度高: 耗时

  2. 一次返回数据过多: 时间损耗在数据组装和网络传输

  3. 操作 bigkey: (分配内存,删除时释放内存)(避免写入、unlink 代替 del、lazy-free del 释放内存放到后台线程执行)

    redis-cli -h 127.0.0.1 -p 6379 --bigkeys -i 0.01 # 扫描大 key,原理是内部 scan, i 参数控制扫描频率,避免影响线上服务

# 存储大 key

如果慢日志中都是复杂度不高的命令可能是写入大 key.写入数据时需要分配内存,删除数据时需要释放内存空间.

  • 业务 : 检查业务代码,评估数据量大小
  • 技术 : reids-cli -h host -p port bigkey -i 0.01 : i 参数控制扫描间隔,单位秒,其实就是内部执行 scan 遍历所有 key,获取长度.只能扫描出元素最多大 key.元素多不一定占用内存多,4.0 版本后可以使用 lazy-free 异步释放大 key

# 集中过期

平时没有延迟,但是某一时刻延迟较大,即在执行主动过期定时任务时需要大量删除过期 key.业务访问必须等待过期任务执行结束.这个并不会记录在慢日志中.

  • 业务 : 代码中查找 expireat、pexpireat 等,然后增加随机过期时间
  • 技术 : info 命令中 expired-keys 代表目前为止,累计删除的过期 key 数量,监控该指标,如果短时间内突增,与延迟时间是否一致

# 实例内存达到上限

当到达 maxmemory 时,写入数据需要先删除一部分数据,即 8 种内存淘汰策略.

lru : 从实例中取出一部分热 key,然后淘汰一个最少访问的,把剩下的存到一个池子中,继续取出一批 key,与之前池子中的 key 比较再淘汰一个最少访问的 key,以此循环,直到内存降到 maxmemory 之下.

# fork 耗时

fork 过程中需要拷贝内存页表给子进程,如果内存过大拷贝时间过长.

可以执行 info 查看最后一次 fork 执行耗时, latest_fork_usec 单位毫秒,备份,主从第一次同步都会 fork.

# 绑定 cpu

绑定 cpu 后进行持久化 fork 出的子进程会继承父进程 cpu 偏好,子进程与父进程发生 cpu 争抢.一般如果开启 rdb、aof 就不要 cpu 绑定.

# 是否使用 swap

检查机器内存情况是否因内存不足使用到了 swap.释放 redis 的 swap 需要重启实例.一般先主从切换,释放旧节点的 swap,重启,待数据同步完成后切换回主节点.

需要对机器内存和 swap 使用情况监控,内存不足和使用 swap 时及时发觉.
